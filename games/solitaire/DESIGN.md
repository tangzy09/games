## 0. 调研结论（v1 的全部依据，证据见 §12）

### 0.1 爽点不是"赢"，是赢之后那段**纸牌瀑布**

玩家原话：9 岁那年赢牌时，**「那些弹跳的牌感觉像全场起立鼓掌」**。Vista 砍掉它引发十几年怨念，还被人反复用 canvas 复刻成梗（Know Your Meme 有词条）。

⇒ **纸牌瀑布是产品的心脏，不是彩蛋**。要长、要过度、要不需要玩家操作。我们的 canvas 引擎做这个几乎零成本（重力 + 0.85 反弹阻尼）。

### 0.2 这个品类最大的怨气是「这局根本赢不了」——而且是**信任问题**，不是数学问题

**硬数据**（JAIR 2025 论文 Solvitaire，烧了 ~30 CPU 年）：

| 玩法 | 理论可解率（完美信息） | 人类实际胜率 |
|---|---|---|
| **Klondike draw-3**（标准） | **81.9%** ⇒ **18% 的局怎么打都赢不了** | **~11%**（1/9） |
| Klondike draw-1 | 90.5% | ~30-40% |
| **FreeCell** | **99.999%**（86 亿局里约 1/84000 无解） | 新手 40%，熟手 75-80% |
| Spider 4 花色 | ≈99%（理论上几乎都可解） | **<10%** |

**关键洞察**：**理论可解 82% 与真人胜率 11% 之间的巨大落差，就是阴谋论的温床。** 玩家不会怪随机数，只会怪你。微软官方论坛上有《Microsoft Solitaire 是不是被操纵了？》的专帖，官方回答「不同模式用不同洗牌算法」反而火上浇油。

⇒ **§2 的差异化直接建在这条上。**

### 0.3 微软纸牌的口碑已经塌了 —— 这是我们的整个市场缺口

玩家实测投诉（微软自家 Q&A 论坛）：
- **「一口气 12 个广告，中间一局都没玩上」**、开局前连播 3 个 30 秒广告
- 广告 X 假关闭、二次弹窗、**绕过静音外放**
- **付了订阅照样出广告**
- **Windows 更新把我 2017 年起的全勤连胜清零了**

它把一个**秒开的离线小工具**做成了联网服务，于是拿到了服务型产品的全部差评（掉线、崩溃、清档、耗电）。

⇒ **它的护城河是分发和怀旧，不是产品质量。** 而这个品类的用户是**高龄 + 低容忍 + 情绪调节导向**（有玩家说广告「破坏了我用纸牌做情绪调节的用途」），恰恰最恨这些。

### 0.4 撤销/提示/重开是「基本人权」，把它们做成激励视频 = 差评的最短路径

玩家原话：**「他们故意让你更容易无路可走，好逼你看广告拿提示、再看广告拿 Joker 续命」**、「每天只给 10 次撤销，多要就得看广告，**别家 Solitaire 根本不需要**」。

⇒ **§7 的红线。**

### 0.5 两条基调（写死不动）

- ⛔ **不要体力，不要押注式金币**。Klondike 品类对此**零容忍**（「纸牌凭什么不让我玩下一局」）；头部产品也基本不敢上。Solitaire Grand Harvest 的押注式金币被骂成「披着纸牌皮的老虎机」（期望值为负：「花 5000 金币玩一局，最多赢回 500」）。
- 📉 **单局 5-15 分钟、单次会话 10-15 分钟、每天 4-6 次**。这是所有品类里 session 最长的之一（远高于手游中位 3-5 分钟），用户偏年长、偏女性、65+ 严重 over-index。**场景是上班摸鱼/睡前/情绪调节。**

## 1. 三种玩法（规则以此为准）

### 1.1 Klondike（经典微软纸牌）
7 列 tableau（1..7 张，仅顶牌明），4 个 foundation（同花 A→K），tableau 交替色降序，**空列只放 K**。翻牌堆 draw-1 / draw-3 可切换。

**标准计分**（Windows 原版）：waste→tableau +5 / →foundation +10 / tableau→foundation +10 / 翻开暗牌 +5 / foundation→tableau **−15** / draw-1 每过一遍牌堆 −100（第一轮后）、draw-3 每次回收 −20。计时模式每 10 秒 −2，通关奖励 `700000 / 秒数`。

### 1.2 FreeCell（空当接龙）
52 张**全明牌**，8 列 + **4 个自由单元** + 4 foundation。空列可放任意牌。

**Supermove 公式**（一次能搬几张，必须实现对）：
```
C = (空 free cell 数 + 1) × 2^(空列数)
移动到空列时： (空 free cell 数 + 1) × 2^(空列数 − 1)   // 目标列不能再当中转
```

**⭐ 微软局号 1:1 复刻**（几乎白送的营销梗）——就是 6 行 LCG：
```
state = (214013 × state + 2531011) mod 2^31 ;  rand = state >> 16   // 0..32767
以局号为种子，从剩余牌堆按 rand % 剩余张数 抽牌交换，横向发进 8 列
```
⇒ 传说中「32000 局里唯一无解的 **#11982**」我们可以**原样提供**。
⚠ **但它和「每一局都存在解法」在同一个 app 里正面打架**（红队 M1）：在一个满屏「✓ 本局存在解法」的世界观里，突然冒出一局「这局无解」，新玩家只会困惑。
⇒ **做成「历史陈列 / 彩蛋」，不是成就**（成就条件写成「你试过了」）。放在公平页旁边，当作「我们连这个都告诉你」的诚实证据。

⚠ **微软局号只有 32000 局（16-bit）** ⇒ 「局号」叙事天然把牌局池限死，重度玩家几年内会撞重复。
⇒ FreeCell 提供两套：「**经典局号 1–32000**」（怀旧 + #11982 梗）+「**无限模式**」（自有 PRNG）。

### 1.3 Spider（蜘蛛纸牌）
**双副牌 104 张**，10 列，stock 每次发 10 张（**发牌前不许有空列**）。tableau 不分花色按点数降序叠放，但**只有同花连续序列**才能整体搬动。凑齐同花 K→A 自动移走，8 组全清即胜。1/2/4 花色三档难度。

**微软计分**：起始 500 分，**每走一步 −1（含 undo）**，每完成一组 +100。

### 1.4 ⚠ 规则的魔鬼细节（不定死这些，实现必错 —— 红队 S12）

**Klondike**（这几条**直接决定可解性**，必须与 solver 的模型一致，见 §2.4）：
- **draw-3 的 waste 回收顺序**：保序 还是 反转？（⚠ 这一条直接改变可解率）
- 翻堆次数有无上限？draw-3 剩 <3 张怎么发？
- 能否从 foundation 取回牌（是，标准计分 −15）。K 序列能否整体搬到空列（是）。
- **autoplay 连锁的 undo 粒度**：一次撤销整串，还是一张一张？

**⚠ autoplay 的「安全收牌」规则有个真 bug**（红队 S12）：
标准写法「一张牌可无脑收 ⟺ 两个异色花色的 foundation 都已到 `rank − 1`」——
**在 rank ≤ 2 时过于保守**：黑 2 收走后**没有任何牌需要它承接**（红 A 直接进 foundation，不需要落在黑 2 上）。
⇒ 必须加 **`rank <= 2 → 无条件安全`**。autoplay 是天天用的功能，这个保守会让玩家觉得「它怎么不收」。

**FreeCell**：
- supermove 的 `空列数`：目标是空列时，**该空列是否计入**？（经典 off-by-one ⇒ **先减再算**，见 §1.2 公式）
- **supermove 是原子操作** ⇒ **undo 必须一步撤销整个 supermove**，绝不能把玩家扔进中间态。

**Spider**：
- 完成 K→A 组**自动移走**（照抄微软）。
- ⚠ **undo 必须能整体回滚两个复合动作**：「发 10 张牌」和「自动移走的一组」。
- 「发牌前不许有空列」**不是死锁**（空列永远能被任意单牌填上），但玩家点 stock 却「什么也没发生」= 经典的「这是不是 bug」投诉
  ⇒ **必须给明确的 UI 反馈**（「先填满空列才能发牌」+ 高亮空列）。
- 计分「每走一步 −1，**含 undo**」与「撤销永远免费」**不冲突**（不花钱，只扣分）—— 但保不保留这条微软行为，**定：保留**（它是原版手感）。

**计分模式**（三选一 + 计时开关）：**Standard**（§1.1）/ **Vegas**（−52 起始、每张收牌 +5、翻堆限次、bankroll 可累计）/ **None**。
⚠ Vegas 的 bankroll 若被激励视频（免开局费）污染，就**不可比** ⇒ **Vegas 分数不进任何榜单/成就**（blockblast 的教训：永久数值加成会毁掉分数可比性）。

## 2. ⭐ 核心差异化：**每一局都存在解法，而且我们证明给你看**

> 与 blockblast 的「预生成块流」是同一条哲学：不辩解，把公平做成**可验证的机制**。
> ⚠ 但措辞必须精确到变态 —— 见 §2.1 的语义红线。**这一节是全作最容易翻车的地方。**

### 2.1 ⛔ 语义红线：「可解」= **透视暗牌**才可解，绝不能说成「你一定能赢」

**红队抓到的语义炸弹**：论文那个 81.9% 是 **thoughtful solitaire（完美信息，看得见全部 52 张牌）** 下的可解率。
而**玩家看不见暗牌**。所以：

| ❌ 绝不能说 | ✅ 只能这么说 |
|---|---|
| 「这局你一定能赢」 | 「**本局存在解法**」（已由求解器证明） |
| 「是你还没找到」 | 「这局从第 12 步之后就没有解了 —— 一键退回去」（**只陈述事实，不指责**） |
| 「保证可解」 | 「已验证：**在知道全部牌的前提下**，本局有解」 |

**为什么这是死线**：玩家在第 12 步时，**在他能看到的信息下根本无法避免那一步**（他不知道暗牌）。
指责一个**信息上不可避免**的错误，会造出一类比「你坑我」更毒的差评：**「它说是我走错，可我根本没法知道」** —— 而且这次是我们主动挑起的。
调研已经预演过这个反噬：MobilityWare 把可解滑杆调到 100% 后，多了一类新差评「我调到 100% 还是一直输，**这设置根本没生效**」。

**公平页必须坦白讲清楚这件事**（而不是藏着）：
> 「本局已由求解器证明**存在解法**——前提是知道全部 52 张牌。你看不见暗牌，所以**运气仍然存在**：即使有解，也可能需要一点点猜。
> 我们能保证的是：**这局不是死局**。我们不能保证你一定赢。
> 顺带一提，我们的**盲打 AI**（和你一样看不见暗牌）在这个难度档的胜率是 **XX%** —— 这就是「有解」和「打得赢」之间的真实距离。」

**⭐ 最后这句是全作最强的一招**：**主动公开「有解」与「人能赢」的落差**。没有任何竞品敢写这个数字。它把一个语义陷阱变成了诚实的护城河。

### 2.2 「这局还有解吗？」——一等公民 UI（不是角标）

局内常驻一个**大按钮**，实时判定**当前局面**（不是初始局面）：

| 判定 | 反馈（措辞见 §2.1） |
|---|---|
| **仍存在解法** | 「这局还有解。」（**只安抚，不剧透走法** —— 剧透了就没得玩了） |
| **已经没有解了** | 「**这局从第 12 步之后就没有解了。**」+ 一键回到**最后一个仍有解的状态** |
| **算不出来** | 「**算不出来。**」⛔ 诚实红线：**绝不把「不知道」说成「有解」**。我们卖的是可验证的诚实，一次含糊全毁。 |

⚠ **这个按钮就是我们对抗 App Store 4.3(a) 的主武器**（§11）：三合一 + 一个小角标，审核员 30 秒里看不见；**一个大按钮 + 「正在证明…」进度动画 + 结论卡片**，他一眼就知道这不是第 10001 个纸牌克隆。**商店截图第 1 张就是它。**

**技术**（三种玩法不能一刀切）：

| 玩法 | 实时判定 | 做法 |
|---|---|---|
| **FreeCell** | ✅ 真·实时（毫秒级） | `fc-solve`（C/MIT，有 emscripten 版）→ WASM。旧基准 **432 局/秒** |
| **Klondike** | ⚠ 多数亚秒，**长尾极重** | `lonelybot`（Rust/MIT/WASM）跑在 **Web Worker** 里 |
| **Spider** | ❌ 不做可解判定 | solver 在 Spider 上力不从心（论文自承）。只给「还有合法移动吗」，并**明说这不是可解性判定** |

**⚠ 别用 150ms 硬超时**（红队 S8）：超时恰恰发生在**最难的局面**上——也就是玩家真正卡住、真正需要这个按钮的时候，**旗舰功能会在最需要它的时刻失效**。
**正解**：solver 跑 **Web Worker**（引擎无 worker 约定 ⇒ **这是新增的引擎契约**，要实测 Capacitor WebView + 门户 iframe），给它 **5-10 秒 + 「正在证明…」进度动画**。**「等 5 秒拿到确定答案」远好于「150ms 后说算不出来」。**

**⚠ 「回到最后一个有解的状态」怎么找**（红队 S8）：天真做法是对撤销栈里每个历史状态各跑一次 solver（几百次 × 秒级 = 不可用）。
正解：**在撤销栈上二分**（log N ≈ 8-10 次调用）。⚠ 但判定是**三值的**（有解/无解/不知道），二分逻辑要专门设计 + 单测（「不知道」时保守地往更早的状态退）。

### 2.3 ⚠ 可解性筛选**只对 Klondike 有意义**（别做无用功）

| 玩法 | 筛不筛 | 为什么 |
|---|---|---|
| **Klondike** | ✅ **必须筛** | 18% 的局天生无解 —— 玩家挫败感的**唯一最大来源** |
| **FreeCell** | ❌ 不筛 | 本来就 99.999% 可解。难的是人不会玩，不是牌不给赢 |
| **Spider** | ❌ 不筛 | 理论 ≈99% 可解，难度全在人。**筛可解性对 Spider 毫无意义** |

⚠ **draw-1 和 draw-3 是两个不同的可解性问题**（红队 F3）：81.9% vs 90.5% 不是同一批局的两个数字 —— **一个 seed 在 draw-3 无解、在 draw-1 可解是常态**。
⇒ **池按 (draw 模式 × 难度) 分片**；**draw 模式是开局前属性、局中不可改**（否则角标失效），放进「新局」面板而不是设置里。

### 2.4 ⭐ 验证：**用我们自己的规则重放 solver 的解**（否则「已验证可解」是系统性谎言）

**红队 F2 —— 这是本设计最大的工程漏洞**：Klondike 的可解性对规则边界**极度敏感**（waste 回收是否保序、翻堆次数上限、draw-3 剩 <3 张怎么发、能否从 foundation 取回…）。
solver 有它自己的建模假设，我们的 `rules-klondike.js` 是自己写的。**只要有一处不一致，池里标「可解」的 seed 在我们的游戏里可能真的无解。**
而「抽 N 个 seed 用 solver 复验」**复验的是 solver 自己**，跟我们的游戏没有半点关系。

**唯一可信的验证**：

1. solver 导出的不是 `{seed, 解长度}`，而是 **`{seed, 完整的 move list}`**。
2. **`tools/verify-deals.js` 用游戏真实的 `rules-klondike.js` 逐步重放这条 move list，必须以胜利结束** —— 否则该 seed **不进包**。
3. **全量跑，不是抽样。** 这是我们承诺的东西，必须每一条都对。
4. 同一约束适用于 §2.2 的实时 WASM 判定：**solver 的规则模型必须和我们的一致**，否则它会对着一个「在我们规则下无解」的局说「仍有解」。

### 2.5 seed 池：**规模和体积要算账**（红队 S3）

- 重度玩家 4-6 局/天 ≈ **2000 局/年** ⇒ 每个难度档至少要**几十万局**才不会撞重复。
- ⚠ 「导出 JSON 打进包」：100 万条 `{seed,len}` 的 JSON ≈ **20MB+**，web 首屏和 app 体积都吃不住。
- **正解**：存**可解 seed 的位图**（1 bit/seed，32M seed 空间 = 4MB）或分档的**二进制密集索引 + 分片懒加载**。
- **牌局编号 = 原始 seed**（编号空间大 ⇒ 感知上无限）。⛔ 别用「#7」这种连号，它会暴露池规模。

**第一次跑管线就把多个指标一起导出**（红队 S2；跑一次是 CPU 小时级，别跑两遍）：
`{seed, 最短解长度, 搜索节点数, 完整 move list, 首步可行分支数}`。
⚠ **「最短解长度」的区分度存疑**（Klondike 赢局必须收满 52 张，最少移动数的下界几乎固定）—— 真正有区分度的可能是**搜索节点数**和**解的鲁棒性**（有几条不同的解 / 首步有几个不劣分支）。所以一次跑全，别发现不够用再重跑。

### 2.6 ⭐ 盲打 AI（`tools/sim-blind.js`）—— 本设计**最重要的缺失工具**

blockblast 的一切数值都靠 `tools/sim.js` 钉死。solitaire 的所有难度假设**目前没有任何校准手段**，而本仓无后端 ⇒ **零埋点**（blockblast 已经吃过这个亏）。

**做一个不透视暗牌的启发式玩家**（翻暗牌 > 空列 > 收 foundation 的贪心 + 有限回溯），跑每个难度档 × N 局，报**盲打胜率**。

它一次性解决三件事：
1. **难度标签有经验依据**（Easy/Normal/Hard 是真实的盲打胜率，不是拍脑袋）
2. **量化 §2.1 的落差**（「有解」vs「人能赢」）—— 这个数字直接印在公平页上，**是全作最强的诚实牌**
3. **回归基线**：改了 `deal`/`rules` 之后不劣化

⚠ 调研里「人类胜率 ~11%」那一列**全部来自无样本量的 SEO 站点**（§12 已自认）。**别把它当调参输入** —— 盲打 AI 是唯一能自产这个数字的办法。

## 3. 纸牌瀑布（产品的心脏，不是彩蛋 —— §0.1）

赢局时全屏纸牌瀑布：每张牌从 foundation 弹出，**重力 + 0.85 反弹阻尼**，在屏幕上撞出**累积的彩虹拖尾**，持续 5-10 秒，**不需要玩家操作**（点一下可跳过；设置里可关）。配一声长的、上扬的胜利音（WebAudio 合成）。

### 3.1 ⚠ 「canvas 做这个几乎零成本」是**错的成本估计**（红队 S9）

**① 拖尾与引擎契约正面冲突。** 经典瀑布的美感来自**永不擦除的累积拖尾**，而引擎契约是 `renderAll()` **每帧全屏重画**。若每帧重画全部历史轨迹：5-10 秒 × 60fps × 52 张牌 ⇒ 每帧几万个圆角矩形，低端安卓 WebView 必挂。

> **正解（要在引擎契约上打一个洞，写进设计）**：
> 一层**持久 trail canvas（永不清）** + 主 canvas 每帧只画运动中的牌 → 最后合成。

**② 52 张牌每帧重画 = 每帧几百次 fillText + 圆角路径**（不止瀑布，**正常对局也是**）。
⇒ **每张牌必须预渲染成离屏 sprite 缓存**（按当前 cardW 缓存一次；换皮肤/换尺寸时重建）。
§10「纯 canvas 画牌不用图片素材」听起来省事，**不做 sprite 缓存就掉帧**。

**③ 状态机与存档**：`G.phase = 'cascade'`；**存档时必须归一化回终局态**（blockblast 踩过「`dragging` 存进档 = 畸形态白屏」的坑）；后台/来电中断、跳过，全部要定义。

**④ `fx.js` 是独立工作项**，不是 §3 这四行（blockblast §8.1 有先例）。

## 4. 入场券（不是加分项，缺了就是一星）

- **无限撤销**（Klondike 玩家把 undo 当成玩法本身：卡住时反复试探别的走法）
- **提示**（枚举所有合法移动 + 启发式排序：翻暗牌 > 空列 > 收 foundation > 移动序列。「保证不走死」的 hint 需要 solver，二期）
- **自动收牌 autoplay**（判定规则：一张牌可以无脑收 ⟺ 两个异色花色的 foundation 都已到 `rank − 1`）
- **重开 / 换一局**（永远免费）
- **统计**（胜率/最快/连胜。玩家会**拿统计数字审判你的公平性**，所以必须准）
- **⚠ 任何新增装饰（横幅/社交/推送）必须可关** —— 玩家原话：「Now you have added all of these bells and whistles that **can't be turned off**」

## 4.5 ⚠ 无限撤销把统计架空了 —— 统计口径必须在 P1 定死（红队 F4）

**三件事互相抵消**：§4 说无限撤销是基本人权（对）；§2.2 说一键回到最后有解状态（很强）；§4 又说「统计必须准，玩家会拿数字审判你」。
⇒ **只要肯撤，任何人 100% 胜率、连胜无限。** 那「胜率」显示的是什么？连胜奖励的是什么？
（blockblast 遇到同构问题时用「用过撤销 → 最高二星」解决了；这里目前**零机制**。）

**定死**（决定 `G` 里存什么计数器，P1 就要有）：

| 口径 | 定义 | 用途 |
|---|---|---|
| **总胜率** | 赢了就算 | 给玩家看「我玩了多少」 |
| **零撤销·零提示胜率** ⭐ | 全程没用过撤销/提示/「还有解吗」 | **这才是玩家会拿去炫的那个**；成就/排行挂它 |
| 放弃 / 重开 | **算输**（写死，别含糊） | 否则玩家会用「重开」刷胜率 |

**每日挑战能否无限重试同一局 —— 必须定死**：
- ❌ 不能重试 ⇒ 连胜纯看运气（Klondike 盲打胜率量级很低），**连胜必断，「补签一次」根本不够**。
- ✅ **可无限重试 + 可撤销**（推荐）⇒ 连胜的实质是「**今天有没有坐下来玩**」—— 这才自洽，也才是可持续的留存机制。**采用这个。**

## 5. 每日挑战 + 连胜（留存三件套，但先把地基做扎实）

- **每日挑战**：三种玩法各一局/天，`seed = hash(日期)` 从可解池取 ⇒ **全球同一局**。
- **连胜 streak** + 月度日历（一整月不断的皇冠）。
- ⚠ **连胜会把你产品的每一个技术缺陷放大成人身伤害**（微软实锤：崩溃/掉线/更新清档 → 玩家 2017 年起的全勤没了）。所以：
  - **本地优先、离线可用、绝不丢档**（这正是微软崩掉的地方 = 我们最好抄的缺口）
  - **连胜给一次「补签」**（断了不至于让人怒删）

## 6. 「秒开 + 离线」是我们的隐性卖点

微软把纸牌做成了联网服务（HN 原话：「没网就是残废，而且**耗电像没有明天**」）。我们是纯 canvas + 本地存档 ⇒ **秒开、离线、不耗电、不丢档**。这条要写进商店描述。

## 7. 变现：**纸牌是最好赚的品类之一**（前提是广告放对位置）

### 7.1 为什么纸牌比消除类好赚 —— session 长度的结构性优势

| | 单次会话 | 每天次数 |
|---|---|---|
| **纸牌** | **10-15 分钟**（所有品类里最长的之一） | 4-6 次 |
| block puzzle（对照） | 3-5 分钟 | 2-3 次 |

**长会话直接改变了变现的最优解**：玩家盯着牌面十几分钟，**横幅就挂十几分钟** —— 曝光时长极高，而且**完全不打断玩法**。这是纸牌相对于消除类的结构性优势，也是我们的主力收入。

### 7.2 收入结构（四条腿）

| 广告位 | 怎么放 | 备注 |
|---|---|---|
| **① 底部横幅（常驻）** | **主力收入**。⚠ **绝不遮牌** —— 牌面布局要为它预留空间（`safeBottom`），而不是盖上去。Spider 的 10 列 + iPhone 窄屏是最紧的情况，必须实测 | 长会话 = 高曝光时长 |
| **② 插屏** | **只在结算页之后**（赢局/放弃），**每 3 局最多一次**。⛔ 绝不局间连播、绝不在「玩到一局之前」播 | 微软的「12 连播」是这个品类最致命的叙事 |
| **③ 激励视频** | **只用于纯增益**：牌背/桌布皮肤、额外的每日挑战、Vegas 模式免掉开局买牌费（−52）。⛔ 绝不用来换撤销/提示/重开 | Vegas 的 −52 是天然的激励接入点，且不违反红线（可选模式 + 纯增益） |
| **④ 一次性去广告 IAP** | **$4.99 买断**（微软是订阅 $9.99/年，而且**付了还出广告** = 它最毒的一条差评） | 见下：这批人正在被逼疯，且有支付能力 |

**⑤ Web 版走游戏门户**（GameDistribution / CrazyGames / Poki）—— 纸牌在门户上表现通常不错，且我们本来就是纯 canvas，接门户 SDK 的成本很低（见 `html5-game-portals` skill）。

### 7.2.1 ⚠ 激励视频**必须有消耗端**，否则第三条腿约等于零（红队 S5）

**算一下现实**：单局 10-15 分钟 + 插屏「每 3 局最多一次」⇒ **一次会话最多 1 次插屏，多数会话 0 次**；激励视频若无人触发，**收入实质只剩横幅一条腿**。
红线（不锁功能）是对的，但**红线只砍掉了强制变现，没有建任何自愿变现的需求侧**。

**解法：把牌背 / 桌布 / 牌面主题做成明确的收集系统**（**20-30 套 + 图鉴**）：
- 解锁三条路：**成就** / **连胜里程碑** / **激励视频** —— 给激励视频一个真正的消耗端。
- 另卖一个「**全解锁**」IAP（与「去广告」分开卖，两个都要）。
- ⭐ **顺手补上 §11 的 4.3 差异层**：一个有图鉴、有收集的纸牌，审核员一眼就知道不是第 10001 个克隆。
- ⛔ 但**皮肤绝不影响玩法**（同 blockblast：不给永久数值加成，否则毁掉分数可比性）。

这是 TriPeaks / Grand Harvest 真正在赚的那套东西的**无毒版**：**只卖装饰，不卖胜利**。

### 7.3 「广告克制」不是慈善，是**差异化 + ASO 关键词**

调研信号很强：**`no ads` 已经是这个品类的可搜索词** —— App Store 上有一整批叫 "Solitaire - No Ads" 的产品，说明存在**一大批愿意为清净付钱的存量用户**。而这批人的画像是：**高龄、有支付能力、对骚扰零容忍**，且正在被微软的 12 连播逼疯。

⇒ 免费用户看横幅 + 温和插屏；受不了的人花 $4.99 一次性买断。**两头都赚，且「不骚扰」本身进关键词。**

### 7.4 ⛔ 红线（都是被差评逼出来的，一条都不能破）

1. **撤销 / 提示 / 重开 / 换一局 —— 永远免费，永远不看广告。** 这是纸牌的基本人权（§0.4）。
   玩家原话：「他们**故意让你更容易无路可走**，好逼你看广告拿提示。」
   **省下那点 eCPM，换来的是「这游戏是骗子」的定性 —— 然后卸载，LTV 归零。**
2. **绝不局间连播插屏，绝不在「玩到一局之前」播广告。**
3. **不要体力，不要押注式金币**（§0.5）。Grand Harvest 的押注式金币被骂成「披着纸牌皮的老虎机」（期望值为负）。
4. **一次性 IAP，不是订阅。** 买了之后**功能不能变少**（激励奖励改为直接给）。
5. **横幅绝不遮牌。** 布局为它留位，不是盖上去 —— 盖住牌面等于偷走一局。

## 7.5 ⭐ 无障碍：这个品类的用户 **65+ 严重 over-index**，而没人为他们设计

调研数据：纸牌用户偏年长、偏女性，**65+ 在休闲卡牌上严重 over-index**。而竞品的 UI 基本是给年轻人做的。这是一块**没人捡的地**，而且成本极低：

- **点击移动（tap-to-move）**：点一张牌 → 高亮所有合法落点 → 点目标位。**不强制拖拽** ——
  手抖 / 关节炎 / 帕金森的用户拖不准，这不是矫情，是他们玩不了。**拖拽和点击两种都要支持，可切换。**
- ⭐ **四色花色牌（four-color deck）** —— 梅花绿 / 方块蓝。**这是主流 solitaire 的标配选项，而我漏了。**
  对老花 / 色觉衰退用户，它的价值**高于大字号**（红黑在小牌面上最难分的就是 ♠/♣ 和 ♥/♦）。成本：一个调色板开关。
- **落点就近吸附 + 容差** —— 手抖用户的实际需求，**比 tap-to-move 更底层**。
- **大字号模式** + **高对比牌面**（点数/花色要在 60px 缩略图下也认得出）。
- **单击自动收牌**（双击送 foundation 是标准做法，但双击对手抖用户也难）。
- 所有装饰（横幅提示、动画、音效）**必须可关** —— 玩家原话：「Now you have added all of these bells and whistles that **can't be turned off**」。

⇒ 这既是善意，也是**差异化**（"the solitaire that actually works for your hands"），还能进 ASO 长尾词（`easy solitaire for seniors` / `large card solitaire`）。

## 7.6 ⚠ 布局：四个约束互相打架，**纸牌 UI 最难的部分就是这里**（红队 S4）

同时成立是不可能的，必须显式取舍：

| 约束 | 来源 |
|---|---|
| Spider **10 列** | §1.3 |
| iPhone SE / 13 mini 竖屏 **CSS 宽 375px** ⇒ 每列 **≤35px** | 现实 |
| 「60px 缩略图下也认得出」+ 大字号高对比 | §10 / §7.5（老年用户） |
| 底部横幅**预留空间不遮牌**（再吃 50-60px 高） | §7.2 |

**⇒ 三套玩法各给一张布局规格表**（P1 就要写，否则 `render.js` 会推倒重写）：
`可用宽高 → cardW / 列间距 / 明牌 offset / 暗牌 offset（两者不同！）/ 最长列压缩策略 / 横幅预留 / safe-area`

- **Klondike 最长列**：6 张暗牌 + K..A 13 张 = **19 张** ⇒ 竖屏高度里必须**动态压缩堆叠 offset**。
- **Spider**：接受小牌 + **只画左上角标**（不画中央花色）；并给「**大牌模式**」作为无障碍出口（横屏 / 横向滚动）。

## 8. 工程（严守引擎契约）

### 8.0 ⚠ 引擎契约的两个坑（红队 S10/S11）

**① `hitTest` 是「后注册优先」（倒序遍历）** —— 我查过 `engine/canvas.js`：`for (let i = hitAreas.length - 1; i >= 0; i--)`。
⇒ **render 必须按绘制顺序（底 → 顶）注册 hit 区**，顶牌才会赢。
**这是一个必然会犯的 bug，且症状诡异**（点一叠牌命中最底下那张）。写死在这里。

**② 别在 P1 把 drag 抽进 engine。**
blockblast 的 `drag.js` 只有 127 行，语义是「**托盘块 → 网格吸附**」（锚点=按住的格、幽灵、非法回弹）。
solitaire 的拖拽是**完全不同的东西**：牌堆父子链、**多张牌整体拖**、落点是**重叠的堆**、还要与 **tap-to-move 共存于同一张牌**（需要 tap/drag 阈值仲裁：位移 <10px 且时长 <250ms = tap）。
两者的公共部分几乎只剩 pointer 事件归一化。而 root CLAUDE.md 明写「**改 engine 牵动 4 个线上游戏**」。
⇒ **P1 先在 `games/solitaire/js/drag.js` 自己写，做完再看能抽什么。**

### 8.1 分层（index.html 加载顺 = 依赖顺）

引擎 → `prng` → `cards`（牌/花色/牌堆）→ `deal`（发牌：Klondike 从可解池 / FreeCell 微软 LCG / Spider 随机）→ `rules-klondike` / `rules-freecell` / `rules-spider`（三套纯规则，各自可单测）→ `core`（选中/移动/撤销栈/计分/胜负）→ `hint` → `storage` → `drag` → `fx`（**纸牌瀑布**）→ `render` → `main`。

- **纯逻辑层**（双导出，node 可单测）：`prng / cards / deal / rules-* / core / hint`。
- **拖拽层复用 blockblast 的 `drag.js` 经验**（引擎的 `input.js` 只有 tap/swipe，没有 drag&drop）。本作是第二个拖拽游戏 ⇒ **该考虑把 drag 抽取进 engine 了**（blockblast 的 DESIGN §5 埋的伏笔）。
- **零硬编码文案**：`T('key')` + `locales/<lang>.json`（**嵌套结构**）。首发 en + zh-CN，代码零硬编码 ⇒ 加语言零改码（根 CLAUDE.md 的语言策略）。
- **存档**：⚠ 别存「当局快照」—— 一局 Klondike 撤销栈可能几百步，每步存整盘 = 存档爆炸；而「恢复后不能撤销」是这个品类的**经典一星**（玩家把 undo 当玩法）。
  **正解：存 `seed + 完整 move list`**（发牌确定 ⇒ 可重放），撤销 = 重放到 n−1 步。天然白送「回放 / 分享解法」，还与「牌局编号」叙事一致。
  ⚠ **前提**：`deal.js` 的洗牌必须是**可复现的纯函数**（由编号确定性还原牌序）—— 写死在 `deal.js` 的契约里，否则 move-list 方案不成立。
  带 `v: N`；**改 `G` 形状必 bump `SAVE_VERSION`**。

### 8.2 测试

- **node 单测**：三套规则各自的合法移动/边界（空列只放 K、supermove 公式、Spider 同花序列搬动、发牌前不许有空列）、撤销精确复原、计分公式、autoplay 安全判定、胜负判定。
- **`tools/verify-deals.js`**：可解池里随机抽 N 个 seed，用 solver 复验「确实可解」——**我们承诺了可解，就必须真的对**（一旦承诺，玩家会把每次失败都当成 bug/欺骗）。
- **E2E**：真实鼠标拖拽（**不用 dispatch 绕过** —— blockblast 那次「假绿」的教训）、赢局瀑布、撤销、每日挑战、三种玩法切换。

## 9. 分阶段（红队 S6：原 P3「FreeCell + Spider」一个阶段**必爆**）

1. **P1 Klondike 可玩**：`cards` + `deal`（可复现纯函数）+ `rules-klondike` + `core`（撤销=move list）+ `drag`（自己写）+ **纸牌瀑布（含 trail canvas + sprite 缓存）** + 撤销/提示/autoplay + **布局规格表** + 统计口径 + 单测 + E2E。
   **验收：赢一局，看那段瀑布，想不想再来一局。**
2. **P2 可解池 + 盲打 AI**：离线 solver 管线（导出 **move list**）+ **`verify-deals.js` 用真实规则重放**（全量）+ seed 池（位图编码）+ 「本局存在解法」角标 + **`sim-blind.js` 盲打胜率** → 难度分档 + **公平页（含「有解 vs 人能赢」的落差数字）**。
3. **P3 「这局还有解吗？」**：Web Worker + WASM solver + 撤销栈二分（三值判定）+ 一等公民 UI（大按钮 + 「正在证明…」）。**这是 4.3 的主武器，不能拖到最后。**
4. **P4 FreeCell**：规则 + supermove（原子 undo）+ 微软 LCG 局号 + #11982 陈列 + 实时判定（fc-solve，毫秒级）。
5. **P5 元层**：每日挑战 + 连胜（口径见 §4.5）+ 统计 + **牌背/桌布收集系统**（§7.2 的激励视频消耗端）+ 成就。
6. **P6 变现/上架**：横幅（不遮牌）+ 插屏 + 激励 + 去广告 IAP + i18n + iOS 壳 + 商店素材。
7. **P7 Spider**（v1.1，**首发可砍**）：⚠ 它是三者里工作量最大的（104 张牌、10 列最难画、同花序列搬动、发 10 张是复合动作、完成组自动移走 —— 后两者的 undo 都要整体回滚），**却是唯一无法兑现「存在解法」这个卖点的玩法**（不筛可解、solver 帮不上忙、人类胜率 <10%）。
   **Klondike + FreeCell 已经覆盖绝大多数用户，且两者都能兑现核心承诺。**

## 10. 美术

纯 canvas 画牌（不用图片素材）：圆角白牌 + 花色符号 + 大号点数，**60px 缩略图下也要认得出**（老年用户是主力人群！**字号要大、对比要高**）。牌背皮肤走 `PAL` 调色板切换（确定性、禁 `Math.random`）。

## 11. ⚠ 上架风险（比 block puzzle 更狠）

**Solitaire 是 App Store 克隆最泛滥的品类之一（几万个）。4.3(a) 风险比 blockblast 更高。**

对策（**P1 就要规划**，不能等到上架前）：
- **「已验证可解 + 牌局编号 + 最短解长度」是审核员 5 秒能看见的东西** ⇒ 商店截图第 1 张就打这个。
- ASO：**别抢 `solitaire` 大词**（底下压着几万个克隆，排不进去 = 零流量）。去吃长尾组合（`winnable solitaire` / `solitaire no ads` / `freecell classic`…）。
  - 💡 **`no ads` 已经成为这个品类的可搜索词**（存在一批「愿意为清净付钱」的存量用户）——这与我们的红线天然一致。
- 图标不能与账号下其他 app 撞（Mando 实锤被拒）。

## 11.5 ⚠ 开工前必须定死的东西（红队 M6/M7 —— 现在全是空的）

- **`G` 的形状**：牌怎么表示（`id 0-51` vs `{suit, rank, faceUp}`）；三合一是**一个 `G`** 还是每玩法一个。
- **`G.phase` 全集**：`menu / playing / selecting / dragging / cascade / win / paused / daily / settings`（blockblast 有，本作还没有）。
- **胜率统计的口径**：见 §4.5（放弃算不算输、重开算不算、撤销后赢算不算 —— 一条都不能含糊）。
- **每日挑战的规则**：能否无限重试（§4.5 已定：可以）；失败算不算断连胜；**三种玩法各一局 ⇒ 连胜是三条独立还是合一**；补签是每月还是终身；**改系统时间只能本地防**（抄 blockblast 的 `daily.js`）。
- **J/Q/K 画什么**：纯 canvas 画不了人像 ⇒ **简化字母 + 花色**，写死。
- 音效表（`GAME_CONFIG.sfx`）· 横竖屏 & iPad · `SAVE_VERSION` 策略。
- ⚠ **i18n 成本翻三倍**：三套玩法的规则说明 + 公平页长文 ⇒ **canvas 需要可滚动长文视图，而引擎没有**（blockblast 已经点过这个缺口，本作因三合一而 ×3）。
